# Claude Agent SDK 系列教程 - 第二章：工具调用与 MCP

这一章开始进入“Agent 真正能干活”的阶段。上一章的 Web 应用已经会聊天了，但还主要停留在生成文本；这一章把工具执行和工具活动可视化接进来，让你第一次看到 Agent 不只是回答问题，而是在完成任务。

## 快速开始

```bash
cp ../.env.local.example ../.env.local
# 在仓库根目录的 .env.local 中填写 ANTHROPIC_API_KEY
# 如需切换模型，可额外配置 ANTHROPIC_MODEL，例如 claude-sonnet-4-6
cd 02-tools-and-mcp
pnpm install
pnpm dev
```

默认情况下，本章会自动复用仓库根目录的 `.env.local`；只有当你想为本章单独覆盖配置时，才需要在 `02-tools-and-mcp/` 目录里额外放一个 `.env.local`。

打开 [http://localhost:3000](http://localhost:3000)。

页面右下角新增了“学习助手”抽屉入口，会按本章的工具调用与 MCP 重点，引导你依次观察真实仓库检查请求、工具活动列表、最终回答与会话对比。

## 这一章解决什么问题

一个只会生成文本的模型，最多算“会说话”；一个能调用工具、读文件、执行动作、再把结果组织回来的系统，才更接近“Agent”。

这一章要解决三个关键问题：

- 为什么 Agent 需要可执行工具，而不只是自然语言推理
- MCP 在教程里的定位是什么，它解决的是“怎么接工具”的哪一层问题
- 当 Agent 开始调用工具后，用户如何看懂它刚刚做了什么

所以本章不仅加入工具调用，也加入了工具活动可视化。

## 这一章的可运行案例是什么

这是一个在第一章基础上继续扩展的 Web Agent 应用，新增了：

- `PromaAgent` 事件驱动封装，把底层 SDK 消息转换为更稳定的 `AgentEvent`
- 工具调用过程的实时展示，包括开始、执行中、完成、失败等状态
- 代码层的 MCP 服务器配置能力，允许在默认工具集之外继续并入 MCP 工具
- 通过仓库根目录 `.env.local` 里的 `ANTHROPIC_MODEL` 切换本章默认模型

你运行后看到的，已经不是单纯的“Claude 回答区”，而是一个能展示 Agent 行动轨迹的界面。

## 动手实践：你应该点什么 / 输入什么 / 观察什么

推荐你这样实验：

1. 输入一个明显需要操作能力的问题，比如 `请查看当前项目根目录有哪些文件，并总结最值得先看的 5 个。`
   观察什么：除了文本回答，中间界面还会出现工具活动记录。

2. 再输入一个更偏工作流的问题，比如 `请先看看 package.json，再告诉我这个项目用了什么主要技术栈。`
   观察什么：Agent 会把“读文件”和“组织答案”串起来，而不是只靠猜测生成。

3. 对照工具活动列表看一次完整流程。
   观察什么：你能看到工具什么时候开始、输入了什么、什么时候完成，以及结果如何被回收进最终回复。

4. 打开代码看 `packages/shared/src/agent/proma-agent.ts` 和 `lib/tool-activity.ts`。
   观察什么：后端并不是把 SDK 原始消息直接硬塞给前端，而是先转换成更适合产品消费的事件层。

工具活动可视化最能帮助你看懂三件事：Agent 为什么会做出某个回答、它中途实际调用了什么工具、一次任务失败或卡住时问题发生在哪一步。

## 这一章对应的 Agent SDK 概念

- Tool use：模型生成的不再只是文本，还会发起工具调用，再消费工具结果继续工作。
- MCP：在这套教程里，MCP 的定位不是“另一个 UI 功能”，而是 Agent 接入外部工具的一种标准化接线方式。
- Agent event abstraction：`PromaAgent` 把 SDK 消息翻译成 `text_delta`、`tool_start`、`tool_result` 等前端更容易理解的事件。

为什么 Agent 需要可执行工具？因为大量真实任务都不是“知道答案”就够了，而是要先去拿信息、操作环境、验证结果，再组织输出。没有工具，Agent 只能描述；有了工具，Agent 才能行动。

## 这一章与 Proma 产品能力的映射

- `PromaAgent` 对应 Proma 产品里把底层 SDK 运行时包装成产品事件流的那一层。
- 工具活动列表，对应 Proma 里“让用户看见 Agent 在做什么”的可观测性能力。
- MCP 配置，对应 Proma 连接外部能力源的扩展机制。

这一章本质上是在搭 Proma 单 Agent 产品里“行动能力 + 过程可见性”的基础设施。

## 学完这一章后你应该掌握什么

- 能解释为什么 Agent 产品必须把工具能力放进核心路径。
- 知道 MCP 在本教程中扮演的是“工具接入标准层”，不是独立炫技点。
- 能读懂工具活动可视化告诉你的信息，并用它分析一次任务执行过程。
- 能理解从 SDK 原始消息到产品事件、再到前端可视化这条抽象链路。
