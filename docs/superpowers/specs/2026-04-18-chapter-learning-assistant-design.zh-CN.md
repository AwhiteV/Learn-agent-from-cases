# 章节学习助手设计方案

**日期：** 2026-04-18

**负责人：** Codex

> 对应英文版位于 `docs/superpowers/specs/2026-04-18-chapter-learning-assistant-design.md`。

## 背景

这个仓库已经把每一章组织成可运行的教程案例，但真正的学习引导目前仍然主要放在各章 README 里。对初学者来说，这意味着他们往往需要在两个上下文之间来回切换，才能真正看懂一章：

1. 正在运行的页面或案例界面
2. 写着推荐操作路径和概念解释的 README

这种分离正好增加了新手的学习摩擦。一个初学者即使把章节跑起来，也仍然可能不知道：

- 第一步该点什么
- 页面上最值得看的区域是哪里
- 某个行为变化到底说明了什么
- 这个现象对应的是哪个 Agent SDK 概念

这次需求的核心，是把一层轻量的“运行时教学引导”直接放进章节页面中，让学习者在案例运行时就能获得推荐操作顺序、观察点和简短的术语解释，同时又不取代原本的章节 UI。

## 目标

1. 为可运行章节增加章节级交互学习引导。
2. 保持章节原本的核心 UI 不变，让学习层辅助案例，而不是盖过案例。
3. 通过一套共享交互骨架，让整套教程更像一个连续系列。
4. 允许不同章节根据概念重点呈现略有差异的教学节奏。
5. 帮助初学者把界面上的可见行为，正确映射到概念、术语和实现文件。
6. 在不牺牲新手体验的前提下，为已经跑通案例的学习者提供一层进阶“实现视角”。

## 非目标

1. 不把每一章做成完整的 LMS 或课程平台。
2. 不用页面内学习助手取代 README 对环境准备和整体说明的作用。
3. 不强迫所有章节使用完全一致的交互脚本。
4. 不做成长篇、阻塞式、会让重复用户厌烦的 onboarding。

## 设计摘要

仓库应增加一套 **章节学习助手**，采用下面的模式：

- 以 **右侧抽屉** 作为主要学习承载面
- 在同一个抽屉内提供 **模式切换**
- 只在关键时刻使用 **页面局部浮层提示**
- 所有章节共享一套学习助手外壳，但每章可以提供自己的学习脚本和少量专属步骤类型

这本质上是一层 **轻量教学层**，而不是一个取代案例本身的新页面。

进一步说，这个抽屉应该同时承载两种学习视角：

- 默认的 **操作引导模式**：面向第一次上手的学习者，强调“下一步做什么、看哪里、为什么重要”。
- 可手动切换的 **实现视角模式**：面向已经跑通过案例、希望理解代码组织、函数职责与数据流转的学习者。

## 核心体验原则

### 1. 引导应该发生在案例运行现场

学习者不应该离开页面，才能知道下一步该怎么学。章节页面本身应该直接告诉他：

- 下一步建议做什么
- 应该看哪块 UI
- 预期会出现什么信号或现象
- 这个现象为什么重要

### 2. 章节页面仍然是主角

学习助手不能长期占据主布局，也不应该在视觉上压过 runnable case。它必须容易打开、关闭、重新开始，也允许被忽略。

### 3. 系列感要稳定，但不能死板

每一章都应该让人感觉它们属于同一套教程，但不能像是把同一个导览反复套壳。稳定的是外壳和使用方式，不是每一章都一模一样的步骤编排。

### 4. 短解释优先于长讲解

每一步尽量只解释一件关键事情。目标不是在当下讲全所有背景，而是让学习者在操作时立刻看懂眼前发生的事。

## 交互模型

### 共享外壳

每个可运行章节都提供一套一致的学习助手外壳，至少包括：

- 一个固定入口，例如 `学习助手`
- 一个右侧抽屉作为主容器
- 一个模式切换控件，例如 `操作引导 / 实现视角`
- 当前章节标题与学习进度
- 步骤切换能力
- `关闭` 与 `重新开始`
- 可选的完成状态提示
- 轻量的术语标签或概念速记

模式切换规则应保持简单稳定：

- 抽屉默认进入 `操作引导`
- 学习者主动切换到 `实现视角`
- 切换模式时，不改变当前步骤位置，只改变这一步骤的讲解方式
- 如果某个章节或某个步骤暂时没有实现视角内容，抽屉应优雅退回到基础模式或显示简短占位说明，而不是暴露空白区域

### 可选的局部浮层提示

局部浮层只在某些步骤确实需要更强“点这里 / 看这里”引导时出现，例如：

- 第一次发送消息时高亮聊天输入框
- 工具调用开始时提示去看工具活动面板
- 权限审批弹出时提示先不要急着 Allow
- 切换 provider 前提示先记录当前执行模式

结构化引导仍然以抽屉为主，浮层只是局部强化，不应该喧宾夺主。

## 信息结构

每一步应该尽量短、稳定、可预测。推荐默认结构是：

1. **去做什么**
   告诉用户当前这一步的具体动作。
2. **看哪里**
   指出当前最值得观察的 UI 区域。
3. **你会看到什么**
   提前说明预期出现的行为或信号。
4. **这说明什么**
   用一句话把它连接回概念。

每一步也可以按需要附带轻量扩展块：

- `术语速记`
- `和上一次结果对比`
- `打开相关实现文件`
- `标记已完成`

### 实现视角的信息结构

实现视角不应该只是把代码文件列表塞进抽屉里，而应该围绕当前步骤建立一条稳定的解释链：

`你刚刚做了什么 -> 页面里发生了什么 -> 内部触发了什么 -> 涉及哪些文件/函数 -> 它们怎么协作 -> 数据怎么流`

推荐默认结构如下：

1. **行为链**
   用一条短链快速说明从用户动作到结果的主路径。
2. **发生了什么**
   用 2 到 4 句解释本步骤内部的因果关系。
3. **看代码**
   给出最关键的文件和函数，并说明它们各自负责什么。
4. **数据流**
   用短链列出输入、状态、SDK 调用和输出之间的流转路径。

为了避免抽屉过满，推荐默认展开 `行为链` 与 `发生了什么`，把 `看代码` 与 `数据流` 设计为折叠区。

## 共享步骤类型

共享外壳应该支持一小组可复用的教学步骤模式：

- **操作步骤**
  学习者需要执行一个具体动作
- **观察步骤**
  学习者需要盯住某个区域看变化
- **对比实验**
  在不同条件下重复同一请求
- **术语解释**
  对当前概念做一句话说明
- **代码映射**
  告诉学习者如果继续深入实现，该看哪个文件
- **阶段总结**
  帮助学习者确认这一章比上一章多了什么能力

每章不必都用到全部类型，但共享外壳应当支持这些常见教学模式。

### 双视角步骤模型

推荐继续使用一套章节脚本来驱动两个模式，而不是为实现视角再创建一套独立系统。也就是说，同一个 step 同时拥有：

- 一份基础 `操作引导` 内容
- 一份可选的 `实现视角` 内容

这样可以保证：

- 同一个用户动作在两种模式下都对应同一个步骤上下文
- 老章节可以逐步补充进阶内容，而不是一次性重写
- 页面、README 与学习脚本的教学顺序保持一致，不会出现“新手模式讲 A，进阶模式突然跳到 B”的割裂

推荐的实现视角字段至少包括：

- `trigger`：用户刚刚做了什么
- `visibleEffect`：页面里出现了什么变化
- `internals`：内部发生了什么
- `files`：应该看哪些关键文件
- `functions`：应该优先理解哪些函数
- `dataFlow`：这一轮输入和输出经过了哪些节点
- `relationships`：这些模块或函数之间是什么关系

## 章节级定制模型

推荐的实现方式是：

- 一套共享学习助手 UI 外壳
- 每章一份章节学习脚本
- 少量可选的章节差异化行为开关或视图集成

这样既能保持系列统一感，也允许每章有不同的教学节奏。

推荐职责划分如下：

- 共享外壳负责：
  - 打开/关闭状态
  - 抽屉布局
  - 步骤渲染
  - 进度管理
  - 通用高亮与浮层提示
- 章节脚本负责：
  - 步骤列表
  - 步骤类型
  - 高亮目标
  - 术语说明
  - 可选代码文件引用
  - 可选实现视角内容
  - 章节结束总结

## 实现视角的维护约束

为了避免实现视角最终膨胀成一套难维护的第二份文档，仓库应遵守以下硬约束：

- 每个进阶步骤只解释一个用户动作，不在同一步里混入多个并行分支
- 文件和函数只列出“理解当前行为所必需的最小集合”，不要把整个工程树铺开
- 推荐控制在每步 `2-4` 个文件、`3-6` 个函数以内
- 不在抽屉里复制 README 的长段背景说明；抽屉负责运行时理解，README 负责章节级叙事
- 第一版解释以手写为主，不依赖自动生成代码讲解，以保证教学准确性

## 分阶段落地建议

不建议一开始同时给 `01` 到 `06` 所有章节补齐实现视角。更适合采用试点式 rollout：

1. 先在 `05-memory-and-skills` 试点
   - 重点解释 memory、skill 与 prompt 组装之间的关系
2. 再在 `06-remote-and-multi-provider` 扩展
   - 重点解释 provider 切换、抽象边界与后端差异
3. 结构稳定后，再补 `02`、`03`、`04`
   - 分别覆盖工具事件链、权限分支和多 agent 协作链路
4. 最后再评估 `01-quick-start`
   - 可以只补轻量代码地图，不一定需要完整进阶模式

## 各章节内容建议

### `01-quick-start`

教学重点：

- 流式输出
- session 持久化
- 新会话与恢复会话的区别
- workspace 感知

推荐步骤：

1. 发送第一条消息
2. 观察回答如何流式增长
3. 看左侧是否出现新 session
4. 新建一个会话并确认上下文被重置
5. 回到旧会话并观察 resume 效果
6. 查看 workspace 文件区域

### `02-tools-and-mcp`

教学重点：

- Agent 不只是回答，而是在行动
- 工具生命周期可见
- MCP 是结构化工具边界

推荐步骤：

1. 提一个需要读文件或查看仓库的问题
2. 观察工具活动面板
3. 看开始、执行中、完成这些状态变化
4. 对照最终回答与工具结果的关系
5. 如有需要，再映射到事件抽象层代码

### `03-agent-with-permission`

教学重点：

- 权限系统会改变执行路径，不只是改文案
- 对比 Allow 和 Deny
- 通过 `AskUserQuestion` 实现结构化人类输入

推荐步骤：

1. 触发一个需要审批的请求
2. 第一次先 Deny
3. 第二次重复同一个请求并 Allow
4. 对比两次执行路径
5. 再做一次 `AskUserQuestion` 场景

### `04-agent-teams`

教学重点：

- 什么情况下任务拆分有价值
- orchestrator 与 subagent 的职责
- resume 与结果汇总

推荐步骤：

1. 提交一个适合拆分的多部分请求
2. 观察主会话是否进入等待 teammate/task 的状态
3. 查看 teammate 列表或任务树
4. 观察任务生命周期事件
5. 判断最终回答是否真的消费了 subagent 结果

### `05-memory-and-skills`

教学重点：

- memory 不等于聊天历史
- skill 模式不等于随机语气变化
- 上下文注入是可观察的

实现视角特别适合强调：

- memory 选择如何进入 prompt 组装
- skill preset 如何改变 system prompt 形态
- 为什么同一个用户问题会因为 injected context 不同而出现回答结构差异

推荐步骤：

1. 创建两条 memory
2. 只注入其中一条并发送问题
3. 保持问题不变，只切换 skill 模式
4. 保持 skill 不变，只切换 memory 组合
5. 删除一条 memory 再重复一次

### `06-remote-and-multi-provider`

教学重点：

- provider 切换时哪些东西会变化
- 哪些边界应该保持稳定
- 为什么 provider abstraction 能保护 UI 层

实现视角特别适合强调：

- 切换 provider 后，哪些 UI 和 transcript 逻辑应该保持不变
- 请求配置如何映射到不同 provider adapter
- provider notes、execution mode 与底层调用边界之间的关系

推荐步骤：

1. 用默认 provider 发送一次消息
2. 切换 provider，用完全相同的消息再发一次
3. 查看 provider 面板
4. 对比 execution mode 和 provider notes
5. 确认请求和响应契约保持稳定

## UI 与行为准则

### 默认存在方式

- 学习助手应该足够显眼，能被快速发现
- 除非某章确实需要，否则不要自动展开成阻塞式 onboarding
- 对重复访问者来说，应该很容易关闭

### 文案语气

- 使用简短、面向新手的语言
- 优先使用“试一下这个”“注意这里的变化”这类表达
- 术语解释尽量压缩到一句话

### 长度控制

- 大多数章节控制在 4 到 6 步
- 每一步尽量只承载一个核心概念
- 正常 walkthrough 里，局部浮层提示控制在 1 到 2 个即可，除非某章明显更适合更多提示

### 可访问性

- 抽屉在桌面端和移动端都应该可用
- 高亮与浮层不能成为唯一的信息承载方式
- 即使不看浮层，用户也应能仅通过抽屉读懂完整步骤

## 内容来源策略

学习助手不应该变成一套脱离章节实际状态的单独文案系统。它的内容必须始终与以下三者保持一致：

- 章节 README 中定义的学习目标
- 章节当前真实可见的 UI
- 当前章节真正强调的 Agent SDK / 产品概念

当章节行为变化时，对应的学习脚本必须与 README、相关 `AGENTS.md` 一起同步更新。

## 建议的文件组织

具体文件名可以按各章情况微调，但整体结构应支持：

- 一套共享学习助手组件或模块
- 每章一份章节学习脚本或配置对象
- 一个小型目标注册机制，用于可选的高亮定位

示意组织方式：

- 共享 assistant UI 放在公共组件区域
- 每章脚本数据放在章节 app code 附近，或章节自己的 learning 模块里

这里真正重要的约束不是目录名，而是把“共享 UI 逻辑”和“章节教学内容”分开维护。

## 实现边界

这个设计故意保持轻量：

- 第一版不要求埋点分析
- 不要求账号级学习进度同步
- 不要求为了教学层单独引入很重的状态机框架
- 不强迫所有章节使用完全相同的提示频率或步骤结构

## 验证标准

如果一个初学者启动任意章节后，能够很快回答下面五个问题，这套设计就算成功：

1. 我第一步应该做什么？
2. 我应该盯住页面的哪一块？
3. 刚刚发生了什么？
4. 这个现象对应哪个概念？
5. 这一章比上一章多了什么能力？

## 风险与缓解

### 风险：学习助手太吵，反而打断体验

缓解方式：

- 以抽屉作为主要结构
- 浮层只少量使用
- 让关闭和重新开始都足够明显

### 风险：所有章节都变得机械相似

缓解方式：

- 统一的是外壳，不是每章完全一样的导览节奏
- 允许章节选择不同的步骤组合与强调点

### 风险：引导文案和真实 UI 渐渐脱节

缓解方式：

- 章节脚本尽量靠近章节代码
- 在更新 README 和章节 `AGENTS.md` 时同步更新学习脚本
- 把学习引导视为 runnable case 的一部分，而不是独立营销文案

## 留给实现规划的问题

1. 第一波实现是直接覆盖 `01` 到 `06`，还是先选 2-3 章做试点？
2. 完成状态要不要按章节存到本地，还是只在当前会话内生效？
3. `代码映射` 链接是否首版就加入，还是等基础引导证明有价值后再补？

## 推荐结论

建议把这项能力实现为一套 **以右侧抽屉为主、按章节配置学习脚本、关键节点辅以局部浮层提示** 的轻量教学层。

这样做能同时得到：

- 一致的教程系列感
- 足够的章节表达灵活性
- 低风险、可渐进上线的实现路径
- 更顺滑地把 runnable case 与概念理解连接起来
