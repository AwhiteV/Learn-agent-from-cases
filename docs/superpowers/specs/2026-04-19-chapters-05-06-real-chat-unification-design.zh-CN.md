# 05-06 章节真实聊天与前端风格统一设计

**日期：** 2026-04-19

**负责人：** Codex

> 对应英文版位于 `docs/superpowers/specs/2026-04-19-chapters-05-06-real-chat-unification-design.md`。

## 背景

当前仓库的 `01-04` 章已经形成了一套相对稳定的学习体验：

- 使用真实的大模型能力，而不是本地拼装的演示回复
- 使用 `session` 维持持续对话
- 使用 SSE 流式渲染回复过程
- 默认复用仓库根目录 `.env.local`
- 页面结构稳定为“左侧历史会话 + 中间聊天工作台 + 右侧章节专属观察面板”

但 `05-memory-and-skills` 与 `06-remote-and-multi-provider` 仍停留在更偏教学演示的形态：

- `05` 通过本地 `chat-engine` 拼装教学回复，而不是真实模型对话
- `06` 通过 provider demo 返回模拟结果，而不是真实模型输出
- 两章虽然有自己的教学面板，但主聊天区的工作方式与前面章节不一致
- 两章都没有对齐 `01-04` 的持续会话工作流与前端工作台风格

这会打断教程作为“连续系列”的学习节奏。学习者在 `01-04` 刚建立起的心智模型，到了 `05-06` 又需要切回另一种交互与实现方式，增加了理解成本。

## 目标

1. 让 `05` 与 `06` 都升级为真实大模型聊天，而不是模拟回复。
2. 让 `05` 与 `06` 都具备和 `01-04` 一致的 `session` 持续对话体验。
3. 让 `05` 与 `06` 的前端主工作台对齐 `01-04` 的三栏式风格。
4. 保留两章各自的教学重点，而不是把它们同质化成完全一样的案例。
5. 默认复用仓库根目录 `.env.local` 作为模型配置来源。
6. 让学习者能清楚看到“统一聊天骨架”之上叠加的章节差异层。

## 非目标

1. 不重写 `01-04` 的架构或视觉系统。
2. 不把 `06` 做成真正的远程基础设施或分布式执行系统。
3. 不在这次改动里引入新的多供应商外部 API 接入。
4. 不删除 `05` 的 memory / skill 教学重点，也不删除 `06` 的 provider abstraction 教学重点。

## 设计摘要

本次改造采用“统一聊天骨架 + 保留章节差异层”的策略：

- 复用 `01-04` 已经验证过的真实聊天工作流
- 把 `05` 与 `06` 的主体验统一成三栏结构
- 把章节差异集中放到右侧工作台与请求组装逻辑中

统一后的结构如下：

- 左侧：`session list`
- 中间：真实流式聊天 transcript
- 右侧：
  - `05` 为 memory、skill 与 prompt preview
  - `06` 为 provider switcher 与 provider inspector

换句话说，学习者从 `01` 走到 `06` 时，应该始终面对相同的“聊天壳子”，但右侧观察面板和后端请求策略会因章节而变化。

## 统一体验原则

### 1. 连续章节应保持稳定的操作框架

学习者在前面章节已经学会：

- 如何开始新对话
- 如何恢复旧会话
- 如何观察流式消息增长
- 如何把一次聊天理解为一个持续推进的任务

`05` 与 `06` 不应要求学习者重新学习另一套聊天交互。章节差异应该体现在“右侧观察什么、后端如何处理”，而不是“聊天骨架完全不同”。

### 2. 教学重点应放在差异层，而不是伪造回复

`05` 要教的是 memory 与 skill 如何影响同一场真实对话。
`06` 要教的是 provider abstraction 如何让同一聊天入口适配不同执行路径。

因此，这两章都应该把真实模型输出作为基础，再通过请求上下文、元数据和侧边观察面板来暴露差异，而不是通过本地模板字符串制造假象。

### 3. 根目录环境配置应继续作为默认入口

为了让整套教程更像一个连续仓库，而不是六个彼此割裂的小项目，`05` 和 `06` 也应像 `01-04` 一样默认读取仓库根目录 `.env.local`。

学习者完成 `01-04` 的环境配置后，继续体验 `05-06` 时不应需要再复制一份独立 env 模板。

## 05 章节设计

### 页面布局

`05-memory-and-skills` 调整为三栏布局：

- 左侧：历史会话列表与新建对话入口
- 中间：真实聊天 transcript、输入框、流式输出状态
- 右侧：
  - `MemoryPanel`
  - `SkillSelector`
  - `Prompt Preview`
  - 与当前回复相关的 memory / skill 说明

右侧工作台保留当前章节的教学重点，但主骨架应与 `01-04` 对齐。

### 真实聊天链路

`05` 的 `POST /api/chat` 不再调用本地 `buildTeachingReply` 生成演示文本，而是改为：

1. 读取用户消息、`sessionId`、`selectedSkillId`、`memoryIds`
2. 根据 `selectedSkillId` 解析出 skill preset
3. 根据 `memoryIds` 读取被激活的 memory 内容
4. 将 skill 说明与 memory block 组装为本轮请求的上下文
5. 调用真实的 Agent SDK
6. 通过 SSE 把流式结果返回前端
7. 将用户消息和助手消息持久化到 session 存储

### skill 与 memory 的教学表达

本章不再通过“不同模板回复”来演示 skill 差异，而是通过真实模型输出中的风格变化来体现：

- `teacher`：偏解释、分步骤讲解
- `builder`：偏方案、偏实施路径
- `reviewer`：偏检查、偏风险与改进建议

memory 的作用也不再是直接拼出可预测的教学句子，而是作为真实上下文被注入给模型。右侧 `Prompt Preview` 继续保留，用于帮助学习者看到上下文是如何被组装的。

### session 持续对话

同一个 skill / memory 组合下，学习者应能连续发送多轮消息并观察：

- 同一个 `session` 如何维持前文上下文
- 切换 skill 后，新一轮请求如何在既有 session 上引入不同系统约束
- 切换 memory 勾选项后，请求上下文如何改变

本章的重点不是“做一个长期记忆系统”，而是“把 memory 当成可选择注入的结构化上下文”。

## 06 章节设计

### 页面布局

`06-remote-and-multi-provider` 同样调整为三栏布局：

- 左侧：历史会话列表与新建对话入口
- 中间：真实聊天 transcript、输入框、流式输出状态
- 右侧：
  - `ProviderSwitcher`
  - `ProviderInspector`
  - 当前 provider 的说明、执行模式、结果元数据

### 真实 provider 路由

`06` 的 `POST /api/chat` 继续保留统一入口，但不再返回纯教学模拟结果。

统一后的处理方式为：

1. 前端始终发送相同契约的请求体
2. 服务端根据 `providerId` 选择 provider 实现
3. 不同 provider 决定“如何执行”与“附带哪些元数据”
4. 实际回复都来自真实模型
5. 结果通过统一 SSE 事件流回到前端
6. session 存储与恢复逻辑保持一致

### local 与 remote-style provider 的边界

本章仍不引入真实的远端基础设施，因此 `remote-style` provider 继续是教学化的“远程风格”实现，而不是生产级远端代理。

但它不应该只是返回假文案。更准确的方式是：

- `local` provider：直接在本地使用 SDK 调用真实模型
- `remote-style` provider：仍在本进程内执行，但在 provider 层体现远程风格边界，例如额外的封装、元数据、延迟模拟或路由说明

这样学习者看到的差异将来自“执行路径与抽象边界”，而不是“一个真、一个假”。

### provider abstraction 的教学表达

本章要让学习者看到两件事：

1. UI 和请求契约可以保持稳定
2. provider 层可以替换执行路径与元数据表达

因此右侧 inspector 应成为主要观察点，而中间聊天区则保持统一体验。

## 共享基础设施设计

### 环境变量

`05` 与 `06` 默认加载仓库根目录 `.env.local`。

默认使用的关键配置包括：

- `ANTHROPIC_API_KEY`
- `ANTHROPIC_MODEL`
- 现有与 debug 或教学演示相关的附加配置

章节内 `.env.local` 可以继续保留为可选覆盖，但不再作为默认文档入口。

### 模型配置

两章都应引入与前面章节一致的模型配置帮助函数：

- 默认模型常量
- `ANTHROPIC_MODEL` 的 trim / fallback 逻辑
- 对应测试覆盖

### session 存储

两章都应补齐或复用与 `01-04` 一致的 session 能力：

- `GET /api/sessions`
- `GET /api/sessions/[id]`
- 可选的删除能力
- `.data/sessions/*.jsonl` 或同等结构化持久化

这样学习者在 `05-06` 里也能体验到完整的恢复工作流，而不是单页内暂存 transcript。

### 流式协议

两章都应采用和前面章节一致的 SSE 工作模式：

- `content` 事件用于增量文本
- `result` 事件用于收尾和 session 元数据
- `error` 事件用于错误回传

如果 `06` 需要 provider 相关的额外事件或元数据，应在不破坏基础契约的前提下追加，而不是改写整套流式协议。

## 前端统一策略

### 与 01-04 对齐的部分

`05` 与 `06` 都应尽量对齐以下体验：

- 左侧会话列表
- 中间滚动消息区
- 输入框与发送按钮位置
- `Session: xxx` 的头部信息
- 空态、错误态与流式中的视觉节奏
- 新建对话与恢复旧对话的交互方式

### 保留章节差异的部分

`05` 与 `06` 不需要完全复制 `01-04` 的右侧面板内容。

应该保留：

- `05` 的 memory / skill / prompt preview 教学面板
- `06` 的 provider switcher / inspector 教学面板

目标是“同一个产品系列里的不同章节”，而不是“同一个页面简单换文案”。

## 风险与权衡

### 1. 真实模型输出会降低教学回复的完全可预测性

这是有意接受的权衡。教学重点将从“固定示例答案”转向“真实请求上下文如何改变模型行为”。

为了减轻不确定性，应保留右侧的可观察上下文信息，例如：

- `05` 的 Prompt Preview
- `06` 的 provider 元数据和执行模式说明

### 2. 复用现有 01-04 聊天骨架会带来一定代码迁移量

这是值得的，因为它能换来：

- 更一致的学习体验
- 更少的环境配置分叉
- 更稳定的 session / SSE 行为

### 3. 06 章节的“remote”表述容易被误解为真实远程执行

因此本章文案必须持续强调：

- 这是 remote-style / remote-like 教学抽象
- 不是生产级分布式基础设施
- 差异重点是 provider abstraction，而不是远端部署本身

## 验收标准

完成后，`05` 与 `06` 应满足：

1. 两章都能使用根目录 `.env.local` 中的模型配置直接运行。
2. 两章都支持新建会话、恢复历史会话、持续多轮对话。
3. 两章都能看到真实模型的流式回复，而不是本地拼装字符串。
4. 两章的前端主布局都与 `01-04` 保持统一的三栏式工作台风格。
5. `05` 仍能清晰展示 memory / skill / prompt preview 的教学重点。
6. `06` 仍能清晰展示 provider switcher / inspector / execution mode 的教学重点。
7. 相关 README、AGENTS、学习助手脚本与测试都保持同步。
